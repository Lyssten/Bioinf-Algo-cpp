# Eulerian Cycle (BA3F)

## Задача

Найти Эйлеров цикл в ориентированном графе.

**Вход:** список смежности ориентированного графа
**Выход:** Эйлеров цикл в виде последовательности вершин

Формат входного файла Rosalind:
```text
0 -> 3
1 -> 0
2 -> 1,6
3 -> 2
4 -> 2
5 -> 4
6 -> 5,8
7 -> 9
8 -> 7
9 -> 6
```

Результат печатается в `stdout` и сохраняется в файл:
- при `--output <path>` — в указанный путь
- при запуске от входного файла без `--output` — в `<input>.out`
- без аргументов — в `eulerian_cycle_output.txt`

**Пример:**
```
Вход:
0 -> 3
1 -> 0
2 -> 1,6
3 -> 2
4 -> 2
5 -> 4
6 -> 5,8
7 -> 9
8 -> 7
9 -> 6

Вывод: 6->8->7->9->6->5->4->2->1->0->3->2->6
```

## Теория

### Задача о Кёнигсбергских мостах

В 1736 году Леонард Эйлер доказал, что невозможно обойти все 7 мостов Кёнигсберга, пройдя по каждому ровно один раз. Так родилась теория графов. Задача о мостах — это задача об **эйлеровом пути**: существует ли маршрут, проходящий через каждое ребро графа ровно один раз?

### Эйлеров цикл vs эйлеров путь

- **Эйлеров путь** — маршрут через все рёбра ровно по одному разу (начало и конец могут различаться)
- **Эйлеров цикл** — то же самое, но начальная и конечная вершины совпадают (замкнутый маршрут)

Не путать с **гамильтоновым** путём/циклом (проходит через каждую *вершину* ровно один раз — NP-полная задача).

### Теорема Эйлера (для ориентированных графов)

Ориентированный граф имеет **эйлеров цикл** тогда и только тогда, когда выполнены оба условия:

1. **Связность:** все вершины с ненулевой степенью лежат в одной компоненте сильной связности
2. **Баланс степеней:** для каждой вершины `in-degree = out-degree` (число входящих рёбер равно числу исходящих)

Для **эйлерова пути** (не цикла) условие ослабляется: допускаются ровно две вершины с дисбалансом — одна с `out - in = 1` (старт) и одна с `in - out = 1` (финиш).

### Алгоритм Хирхольцера (Hierholzer, 1873)

Классический алгоритм — двухфазный:
1. Стартуем из любой вершины, идём по непосещённым рёбрам, пока не вернёмся в начало → получаем начальный цикл
2. Если остались непройденные рёбра, находим вершину на цикле с непройденными рёбрами, строим из неё подцикл и вставляем его в основной

Стековая реализация (используемая здесь) элегантно объединяет обе фазы:
- Идём вперёд, пока есть непройденные рёбра — кладём вершины на стек
- Когда рёбер нет — снимаем вершину со стека и добавляем в результат
- В конце разворачиваем результат

Эта реализация автоматически «вшивает» подциклы в нужные позиции, потому что при обнаружении ответвления мы сначала полностью проходим подцикл и только потом возвращаемся к основному маршруту.

### Биологический смысл

Эйлеров цикл — финальный шаг в конвейере сборки генома:
1. Секвенатор → k-меры (BA3A)
2. k-меры → граф де Брёйна (BA3D/BA3E)
3. Граф де Брёйна → **эйлеров путь** → реконструированный геном

Если геном кольцевой (бактерии, митохондриальная ДНК, плазмиды) — ищем эйлеров **цикл**. Если линейный (хромосомы эукариот) — эйлеров **путь**.

На практике граф де Брёйна реального генома не имеет единственного эйлерова пути из-за повторов, ошибок секвенирования и неполного покрытия. Ассемблеры разбивают граф на подграфы (**контиги**), для каждого из которых эйлеров путь существует.

### Применения за пределами биологии

- **Задача китайского почтальона (Chinese Postman Problem):** почтальону нужно обойти все улицы района, минимизируя повторные проходы. Если граф эйлеров — оптимальный маршрут = эйлеров цикл (каждая улица пройдена ровно один раз). Аналогично — маршруты снегоуборочных машин и мусоровозов
- **Тестирование печатных плат:** робот должен проверить каждое соединение (ребро) на плате ровно один раз. Эйлеров цикл даёт оптимальный маршрут тестирования
- **Рисование фигур одним росчерком:** классическая головоломка «нарисовать фигуру, не отрывая карандаша и не проходя по линии дважды» — это поиск эйлерова пути
- **Генерация последовательностей де Брёйна:** эйлеров цикл в графе де Брёйна порядка n даёт кратчайшую циклическую строку, содержащую все n-граммы. Используется для генерации PIN-кодов в тестировании банковских систем
- **Сетевая маршрутизация:** в SDN (Software-Defined Networking) эйлеровы пути используются для мониторинга сети — каждый канал связи проверяется ровно один раз за обход

## Алгоритм

### Псевдокод

```
EulerianCycle(graph):
    edgeIndex = { v: 0 для каждой вершины v }
    stack = [startNode]
    cycle = []

    while stack не пуст:
        v = stack.back()
        if edgeIndex[v] < |graph[v]|:
            u = graph[v][edgeIndex[v]]
            edgeIndex[v] += 1
            stack.push(u)
        else:
            cycle.push(v)
            stack.pop()

    reverse(cycle)
    return cycle
```

**Ключевая идея:** когда у вершины закончились непройденные рёбра, она добавляется в результат (в обратном порядке). Это автоматически «вшивает» подциклы в правильные позиции.

## Реализация на C++

```cpp
std::vector<int> eulerianCycle(std::map<int, std::vector<int>>& graph) {
    std::map<int, size_t> edgeIndex;                    // (1)
    for (auto& [node, neighbors] : graph) {
        edgeIndex[node] = 0;
    }

    int startNode = graph.begin()->first;               // (2)
    std::vector<int> stack;
    std::vector<int> cycle;

    stack.push_back(startNode);

    while (!stack.empty()) {
        int v = stack.back();                           // (3)
        if (edgeIndex[v] < graph[v].size()) {
            int u = graph[v][edgeIndex[v]];             // (4)
            edgeIndex[v]++;
            stack.push_back(u);
        } else {
            cycle.push_back(v);                         // (5)
            stack.pop_back();
        }
    }

    std::reverse(cycle.begin(), cycle.end());           // (6)
    return cycle;
}
```

### Особенности реализации

**1. `std::map<int, size_t> edgeIndex` — индекс следующего непройденного ребра**
```cpp
std::map<int, size_t> edgeIndex;
for (auto& [node, neighbors] : graph) {
    edgeIndex[node] = 0;
}
```
- `std::map<int, size_t>` — упорядоченный словарь: вершина -> индекс текущего ребра
- `edgeIndex[v]` хранит, сколько рёбер из вершины `v` уже пройдено
- Вместо удаления рёбер из списка смежности (дорого) мы просто увеличиваем индекс — O(1)

**2. `auto& [node, neighbors]` — structured bindings (C++17)**
```cpp
for (auto& [node, neighbors] : graph) {
```
- Деструктурирует пару `std::pair<const int, std::vector<int>>` из `std::map`
- `node` — ключ (номер вершины), `neighbors` — значение (список смежности)
- `auto&` — по ссылке, без копирования

**3. `stack.back()` — доступ к вершине стека без удаления**
```cpp
int v = stack.back();
```
- `back()` возвращает ссылку на последний элемент `std::vector`
- Мы смотрим на текущую вершину, но пока не убираем её — она может ещё иметь непройденные рёбра
- В отличие от `std::stack`, `std::vector` позволяет и `back()`, и индексный доступ

**4. Продвижение по ребру**
```cpp
int u = graph[v][edgeIndex[v]];
edgeIndex[v]++;
stack.push_back(u);
```
- `graph[v]` — вектор соседей вершины `v`
- `edgeIndex[v]` — индекс следующего непройденного ребра
- После взятия ребра инкрементируем индекс, переходим в соседа

**5. Добавление в результат при возврате**
```cpp
cycle.push_back(v);
stack.pop_back();
```
- Если все рёбра из `v` пройдены, `v` фиксируется в цикле
- `pop_back()` удаляет последний элемент вектора (O(1))
- Вершины добавляются в обратном порядке — поэтому нужен `reverse` в конце

**6. `std::reverse` — разворот результата**
```cpp
std::reverse(cycle.begin(), cycle.end());
```
- Алгоритм из `<algorithm>`, принимает пару итераторов
- Работает in-place за O(n) без дополнительной памяти
- Необходим, потому что вершины добавлялись при «разматывании» стека — в обратном порядке обхода

## Сложность

### Время: O(|E|)
- Каждое ребро графа обрабатывается ровно один раз (продвижение `edgeIndex`)
- Каждая вершина добавляется в стек и извлекается из стека суммарно O(|E|) раз
- `std::reverse` — O(|V|), что не превышает O(|E|) для связного графа

### Память: O(|V| + |E|)
- Граф хранится как список смежности: O(|V| + |E|)
- Стек и результирующий цикл: O(|E|) в худшем случае
- `edgeIndex`: O(|V|)
